# project-x-ray

Deep-scan a codebase and write a structured `PROJECT_ANALYSIS.md` — the file a newcomer (human or agent) would otherwise spend an hour assembling: what the project does, how it's built, where everything lives, what conventions the team follows.

## What it does

One command triggers a six-phase scan: directory structure, dependency inventory, architecture mapping, convention extraction, domain summary, and consumption guidance. Output lands in a single markdown file at the repo root. Subsequent agent sessions load that file instead of re-scanning — context reuse instead of context waste.

## When to use

- Starting work in an unfamiliar codebase.
- Onboarding a new team member or agent.
- Snapshotting project state before a large refactor or handoff.
- Any time someone asks "what does this project do?"

## How to invoke

```
analyze this project
x-ray the codebase
write a project overview
onboard me
what does this project do?
map this repo
```

## Example output (abbreviated)

```markdown
# Project Analysis: acme-api

> Generated by project-x-ray · 2025-07-01

## Overview

Acme API is a REST service that manages widget inventory for warehouse
operations. Built on FastAPI with SQLAlchemy ORM, deployed as a Docker
container behind an nginx reverse proxy...

## Directory Structure

├── src/
│   ├── api/          # Route handlers, grouped by resource
│   ├── services/     # Business logic layer
│   ├── models/       # SQLAlchemy models
│   └── core/         # Config, auth, middleware
├── tests/            # pytest, mirrors src/ structure
├── alembic/          # Database migrations
...

## Architecture

Layered: router → handler → service → repository → Postgres.
Services enforce business rules; handlers do request/response only...
```

## Budget

Tree depth capped at 3 levels. Output under ~2,000 lines. Large monorepos get a top-level scan with per-package deep scans offered as follow-ups.
